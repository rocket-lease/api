# Rocket Lease — Cross-repo Conventions

What lives in this file: the handful of rules that need to be consistent **between api, web and contracts** to keep them inter-operable. Everything else (TS strictness, lint, formatter, coverage thresholds) lives in each repo's own config and does not need a shared doc.

## Commits

All commits and PR titles follow **Conventional Commits**: `<type>(<scope>): <subject>`.

Allowed types: `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`.

Examples:

- `feat(reservations): apply EXCLUDE constraint on overlap`
- `fix(auth): refresh token race on app cold start`
- `chore(deps): bump prisma to 6.20.0`

`--no-verify` is forbidden — fix the issue, do not skip the hook.

## Error format (api → web)

All api error responses follow RFC 7807 Problem Details:

```json
{
  "type": "https://rocket-lease.com/errors/vehicle-not-available",
  "title": "Vehicle not available",
  "status": 409,
  "code": "VEHICLE_NOT_AVAILABLE",
  "detail": "El vehículo solicitado ya está reservado para esas fechas.",
  "instance": "/reservations"
}
```

`code` is the enum exported from the `contracts` package and the only field web uses for branching. `detail` is the user-facing Spanish message in v1 (no i18n yet).

NestJS exception filter (`AppExceptionFilter`) standardizes the shape; never throw `new HttpException(...)` with a custom body.

## Date, money, IDs

- **Timestamps**: UTC in DB + API JSON. Web renders in `America/Argentina/Buenos_Aires` via `date-fns-tz`. Reservations use Postgres `tstzrange`.
- **Money**: integer cents in DB and API JSON. `currency` field with ISO 4217 code (`ARS` v1). Display via `Intl.NumberFormat('es-AR', { style: 'currency', currency: 'ARS' })`. No floats anywhere.
- **IDs**: UUID v7 (sortable by creation time) for new entities. Generated by api at insert time, never client-side.

## Domain language

- Spanish canonical for product/UX; English code identifiers. See `CONTEXT.md` for the glossary.
- User-facing strings hardcoded in Spanish (rioplatense), centralized per repo (`src/i18n/es.ts`).
- New domain terms must be added to `CONTEXT.md` in the same PR that introduces them.

## Reservation concurrency

See ADR-0004 for the full design (EXCLUDE constraint + 10-min hold + idempotency keys). The contract: status transitions only via `ReservationsService` in api; web never assumes a transition succeeded until the api confirms.

## Branching

- Branches: `feature/<ticket-slug>`, `fix/<slug>`, `chore/<slug>`, `docs/<slug>`, `refactor/<slug>`, `hotfix/<slug>`.
- Flow: `feature/*` → PR → `dev` → PR → `main`. Hotfix: `hotfix/*` → PR → `main` and back-merge to `dev`.
- Squash-merge only.
- When a US touches contracts + a consumer, create the **same branch name** in both repos. CI in api/web looks for that branch in contracts; falls back to the PR's base ref if absent. See ADR-0007.
